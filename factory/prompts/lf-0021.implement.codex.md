        # Grill Gate

- Owner: repository operator and future service integrators accepting visual baselines outside the current browser session.
- Problem: external images may be untrusted, multi-frame behavior is undefined, and Pillow's exact pixel engine must not be stretched into unsupported perceptual or semantic-quality claims.
- Out of scope: accepting network URLs directly, video/GIF timeline analysis, implementing SSIM inside UIdetox, automatic aesthetic scoring, and adding a perceptual dependency without demonstrated need.
- Review failure: untrusted inputs execute in the main long-lived process when isolation is requested, resource limits are bypassable, multi-frame files are partially compared, subprocess failures lose evidence, or documentation implies Pillow provides perceptual quality assessment.
- Riskiest assumption: an optional constrained worker boundary is sufficient for local external-baseline workflows while deployment-specific OS sandboxing remains the integrator's responsibility.
- Smallest acceptable: external-baseline mode has an isolated worker protocol with time/output/resource bounds, unsupported animation is rejected explicitly, and capability documentation/tests state when a separate perceptual library would be required.
- Recommended choice accepted: no new perceptual dependency now; exact masked RGB evidence is the supported contract until a measured use case proves it insufficient.

# Context

The current product captures local browser PNGs. P3 recommendations are conditional hardening for external baselines and explicit scope closure, not permission to advertise Pillow as an SSIM or semantic-quality engine.

# Acceptance Criteria

- Visual evidence can opt into an isolated worker for external/local baseline processing via a structured request/response protocol; no shell command strings or pickle transport are used.
- The worker enforces request size, path policy, pixel/frame limits, wall timeout, bounded stdout/stderr, deterministic environment, and nonzero/error reporting.
- Main-process validation still treats worker output as untrusted and validates schema version, paths, hashes, numeric ranges, and artifact containment.
- Platform limitations are explicit: the worker is process isolation, not a complete OS sandbox; service deployments accepting uploads require deployment-level sandboxing.
- Multi-frame/APNG/GIF/TIFF inputs are rejected with actionable evidence unless animation support is deliberately added in a future spec.
- URL fetching is not supported by the image worker.
- A capability decision record states that Pillow supplies exact/masked comparison and evidence preparation but not SSIM/LPIPS or semantic design quality; no perceptual library is added without benchmarks and acceptance thresholds.
- CLI/help/docs expose external-baseline isolation, safety limits, and unsupported inputs accurately.
- Tests cover worker success, corrupt request, timeout, oversized output/input, path escape, tampered manifest, missing Pillow, and multi-frame rejection.
- Final full regression and live full-stack smoke pass with both in-process local capture and isolated external-baseline comparison.

# Constraints

- Never claim subprocess isolation alone makes arbitrary uploaded images safe.
- Never deserialize executable Python objects.
- Never read paths outside explicitly allowed roots.
- Preserve the no-network image-processing boundary.
