        # Grill Gate

- Owner: UIdetox maintainers, project operators, and the coding agents receiving the generated handoff.
- Problem: setup stores only the latest effective intent in config; there is no immutable onboarding intent event, no source-linked handoff artifact, and no copy-ready prompt preserving provenance and contracts.
- Out of scope: changing redesign ranking semantics, storing secrets, treating generated prompt text as a source of truth, or extending the current rolling memory log.
- Review failure: intent metadata drifts between config/log/prompt, hostile text escapes the untrusted-data boundary, confirmations overwrite history, writes are non-atomic, secrets are logged, the prompt cannot identify its source event, or existing intent behavior regresses.
- Riskiest assumption: a single canonical event can update the current intent projection, append an audit record, and generate a safe handoff without coupling to the dirty memory Module.
- Smallest acceptable: a deep intent journal/handoff Module atomically records structured intent provenance, renders a prompt through existing prompt-safety rules, integrates with setup/onboarding, and makes the saved event inspectable.

# Context

`DesignIntent` already stores field-level value, provenance, evidence, confidence, confirmation state, and confirmation time. `uidetox intent` inspects current effective intent. `history.save_run_snapshot` is scan-oriented and high-blast-radius; `memory.log_progress` is a rolling free-text log currently being changed by the Chroma-removal work. A separate append-only onboarding journal provides Locality and avoids collision. The generated prompt must embed user/repository text as untrusted data.

# Acceptance Criteria

- A new intent journal/handoff Module creates a versioned immutable event containing normalized intent fields, per-field provenance/evidence/confidence, confirmation status/time, source, project-relative context, and a stable event identifier/fingerprint.
- Events append atomically under `.uidetox/logs/` without overwriting earlier confirmations; malformed existing records fail closed without losing prior data.
- Current config remains the effective intent projection and stays backward-compatible with existing `DesignIntent` readers.
- The agent handoff is derived from the saved event, names the event identifier, includes explicit preserve/constraint contracts, and gives the agent a concrete UIdetox scan/map/redesign workflow.
- All user/repository-authored intent text is serialized through `render_untrusted_data` or an equivalent fixed delimiter; it cannot become agent instructions outside that boundary.
- Prompt output is saved under `.uidetox/` and printed as a copy-ready final onboarding result.
- `uidetox setup` records an event and refreshes the handoff whenever explicit intent is confirmed or updated.
- The onboarding `intent` step runs the existing interactive intent capture, then the `handoff` step records the event and artifact; successful completion marks onboarding complete.
- `uidetox intent` exposes the latest event/handoff reference without changing existing JSON keys incompatibly.
- Tests cover multiple confirmations, deterministic identifiers, hostile multiline/delimiter text, redaction of secret-like fields, atomic failure, malformed journal, event-to-prompt linkage, config compatibility, onboarding completion, and independent `uidetox setup`.

# Constraints

- Do not write intent or prompts into `uidetox/memory.py`.
- Do not include environment variables, credentials, tokens, or full arbitrary config in logs/prompts.
- Do not execute the generated prompt.
- Do not modify scan-oriented run-history schema unless required by a backward-compatible reference field.
- Preserve explicit > mapped > fallback intent precedence.
